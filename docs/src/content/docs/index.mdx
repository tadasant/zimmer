Zimmer is open-source, standards-built, and made for a single circle of trust — you,
or a couple, or business partners. You stay in control of what runs and what merges;
Zimmer handles the toil of getting an agent from a prompt to a verified PR.

## What it does

<CardGrid>
  <Card title="It runs a real agent, headless" icon="server">
    A session is a real Claude Code or Codex process running against a real clone of a
    real repo. It commits, it pushes, it opens a PR, and it hands the result back for you
    to approve.
  </Card>
  <Card title="It has a lifecycle" icon="random">
    `waiting → running → needs_input → archived`. Every session sits in one of those
    states, and the state machine decides the next move, not a human babysitting a terminal.
  </Card>
  <Card title="It wires the context" icon="puzzle">
    Skills, MCP servers, agent roots, hooks, plugins, and references are resolved from a
    catalog by [AIR](/air/overview/) and injected into the clone before the agent starts.
  </Card>
  <Card title="It tells you what's broken" icon="warning">
    These docs include a [Known limitations](/limitations/) page that names real bugs in
    real files. It is meant to be read, not skipped.
  </Card>
</CardGrid>

<Aside type="tip" title="These docs are blunt on purpose">
The premise is that you can read this site instead of the code. That only works if it tells
you what the code would, so it names Zimmer's real bugs and limitations — inline where they
bite, and aggregated on [Known limitations](/limitations/). Where the code and the old docs
disagreed, the code wins and the contradiction is written down. Where the code was genuinely
ambiguous, you get an "unclear — needs confirmation" note instead of a guess.
</Aside>
