## Start here

<CardGrid stagger>
  <Card title="It runs agents, not chats" icon="rocket">
    A Zimmer session is a real coding agent process — Claude Code or Codex — running
    headless against a real clone of a real repo. It commits, it pushes, it opens PRs.
  </Card>
  <Card title="It has a lifecycle" icon="seti:config">
    `waiting → running → needs_input → archived`. Every session is somewhere in that
    machine, and the machine — not a human watching a terminal — decides what happens next.
  </Card>
  <Card title="It wires the context" icon="puzzle">
    Skills, MCP servers, agent roots, hooks, plugins, and references are resolved from a
    catalog by [AIR](/air/overview/) and injected into the clone before the agent starts.
  </Card>
  <Card title="It tells you what's broken" icon="warning">
    These docs include a [Known limitations](/limitations/) page that names real bugs in
    real files. It is meant to be read, not skipped.
  </Card>
</CardGrid>

<Aside type="caution" title="These docs are deliberately blunt">
Zimmer is a single-operator, self-hosted tool with rough edges, and this site names them —
inline, on the page where they bite, and aggregated on [Known limitations](/limitations/).
Where the code and the old docs disagreed, **the code won**, and the contradiction is
written down. Where the code was genuinely ambiguous, you'll see an
"unclear / needs confirmation" callout rather than a confident guess.
</Aside>
