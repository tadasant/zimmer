Claude Code Auth & Account Rotation

How Agent Orchestrator keeps a pool of Claude Code (Max subscription) accounts authenticated, rotates between them on quota limits, and survives the moving target that is Claude Code's OAuth implementation.

Mental model Container topology Data model Invariants Lifecycle Rotation Login Failure modes

1 · The mental model in three sentences

  1. The database is the source of truth. A pool of ClaudeAccount rows holds each account's OAuth tokens; exactly one row per runtime is is_current. The filesystem is derived state written before each session spawn.
  2. The Claude CLI reads two files — an identity file and a tokens file — and in production those two files live in different scopes (one per-container, one shared). Keeping them consistent across containers is the whole ballgame.
  3. Tokens rotate and die. Anthropic invalidates a refresh token the instant it's used; a credential set without a live refresh token is unrecoverable. Every persist path therefore refuses incomplete credentials, and a shared owner marker records whose tokens are on disk.

2 · Container topology — the load-bearing diagram

This is the single most important and least obvious fact in the system. In production, two containers (web + worker) run on one host. The Claude credential tokens file is on a shared bind mount; the identity file is container-local.

The trap: code that reads the per-container ~/.claude.json to answer "whose tokens are in the shared .credentials.json?" gets a confidently wrong answer on whichever container's identity happens to be stale. On the diagram above, web thinks the shared tokens belong to tadas412 while worker thinks they belong to bob — but there is only one tokens file. Trusting the local identity file is how one account's tokens get grafted onto another account's DB row.
The fix: the shared owner marker .ao-credentials-owner.json is written by AO every time it writes the tokens file. It is the authoritative answer to "whose tokens are on disk," and because it is in the shared mount, both containers agree.

4 · The invariants that keep the pool alive

#InvariantEnforced by
I1 Completeness. Never persist a Claude credential set missing accessToken or refreshToken — to the DB or to disk. A refresh-token-less set is unrecoverable. ClaudeAccount.complete_claude_oauth? gating sync_from_filesystem!, sync_tokens_from_filesystem!, write_credentials_to_filesystem!, write_config!
I2 Marker-as-oracle. "Whose tokens are in the shared .credentials.json?" is answered by the shared owner marker, never by the per-container ~/.claude.json (legacy fallback only while no marker exists yet). ClaudeAccount.credentials_owner_email; filesystem_credentials_owned_by_self?; capture_outgoing_filesystem_tokens
I3 DB is source of truth for is_current. The filesystem is written from the DB before each spawn — not the reverse. Filesystem identity is only adopted when a shared-mount timestamp proves an external write actually happened. AccountRotationService#ensure_active_account!; reconcile_with_filesystem! + credentials_changed_externally?
I4 Validate before activate. A candidate account's refresh token is probed against Anthropic's OAuth endpoint before its credentials are written to disk, so a dead account is skipped rather than bricking the live file. AccountRotationService#activate_next_account

5 · Credential lifecycle

How a token travels from a browser login to an authenticated session, and how mid-session CLI rotations are recaptured.

login / capture AO write (stamps marker) external CLI write periodic refresh

The loop on the right is the subtle part: the CLI refreshes tokens on its own mid-session and writes them to the shared file without telling AO. AO recaptures them (gated by the marker, I2) so its DB copy never goes stale and replays a dead refresh token.

6 · Rotation on quota

A running session hits a usage limit. Claude prints You've hit your limit · resets …; ApiErrorRetryService classifies it as :quota_exceeded.
ProcessLifecycleManager calls RuntimeAuthProvider.for(runtime).rotate_for_quota! → AccountRotationService#rotate!.
Capture & retire the outgoing account: sync any CLI-rotated tokens back to its DB row, snapshot its quota, mark it quota_exceeded.
Pick the next candidate (available, lowest priority, excluding the one we just left).
Validate before activating (I4): probe its refresh token against Anthropic's OAuth endpoint. If it fails, skip to the next candidate — do not mark it broken (that previously cascaded and bricked the whole pool).
Activate: capture the outgoing filesystem owner's tokens (by marker), write the new account's config to disk + stamp the marker, mark it is_current, snapshot. Log an AccountRotationEvent.
If no candidate validates → return no_available_accounts; the session pauses with a clear message rather than looping.
Auto-recovery: a quota_exceeded account is restored to active once its reset window passes (quota checks). A needs_reauth account is retried by the refresh cron and, failing that, requires an interactive re-login.

7 · UI-driven login (re-auth)

The only supported way to mint fresh tokens. It never touches the live credential files until the very end.

User clicks Authenticate on /quotas (web). A RuntimeLoginAttempt row is created; RuntimeLoginJob is enqueued (worker).
The worker spawns claude auth login --claudeai under a PTY with CLAUDE_CONFIG_DIR pointed at a throwaway scratch dir — so an in-progress login can't disturb live credentials.
The CLI prints an authorize URL → written to the attempt row → surfaced in the UI. User authorizes in a browser as the target account.
User pastes the <code>#<state> back; the web controller writes it to the attempt row; the worker feeds it to the held-open CLI's stdin (read uncached across the process boundary).
The CLI writes credentials into the scratch dir. The driver captures them into the account's DB row — rejecting an identity mismatch or an incomplete token set (I1).
User clicks Switch → activate! writes the new config to disk + stamps the marker, and marks the row current.

8 · Failure modes & how the design prevents them

FailureWhat happened (2026-06-11)Prevented by
Cross-account contamination A sync path validated the shared credentials against the container-local ~/.claude.json; on the wrong container it grafted one account's access token onto another account's DB row. I2 — marker is the ownership oracle; sync only to the marker's account.
Refresh token erased sync_from_filesystem! & the disk writers persisted a refresh-token-less credential set the CLI had written, deleting the only recovery path. I1 — completeness guard on every read/write of credentials.
Unrecoverable pool Once the surviving refresh token was consumed (Anthropic rotates+invalidates on use), no live refresh token existed anywhere → every call 401'd, and re-login was immediately re-clobbered. I1 + I2 together: no incomplete set persists, and a re-login's marker stops the cross-container clobber.
Rotation cascade (earlier incident) marking each failed candidate needs_reauth during rotation bricked the entire pool on one bad probe. I4 — validation skips bad candidates without marking them broken.
When Claude Code changes underneath us: the file shapes, endpoints, refresh semantics, and login command this design relies on are an undocumented moving target. They are written down in docs/CLAUDE_CODE_OAUTH_ASSUMPTIONS.md. If you find the CLI behaving differently, update that file in the same PR as your fix — that is the durable record that keeps the next outage short.
